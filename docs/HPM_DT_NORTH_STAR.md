# HPM-DT North Star

## 平台全称

HPM-DT（High-Power Microwave Digital Twin）

面向高功率微波系统与效应研究的全中文开源任务级数字孪生 CAE 平台。

## 最高层目标

HPM-DT 的最终任务不是开发一个 Python 项目，也不是完成某一个论文阶段版本，而是持续演进为一个面向高功率微波相控阵、效应分析与数字孪生研究的科研级 CAE 平台。所有版本开发都必须服务于这个长期目标，而不是孤立地增加功能。

平台长期对标对象不是单篇论文或脚本集合，而是下列能力的组合：

- MATLAB Phased Array Toolbox 的阵列算法能力；
- COMSOL App 的工程化交互建模能力；
- CST/HFSS 后处理与结果管理能力；
- 数字孪生平台的状态、数据、验证与审计能力；
- 科研论文流水线的实验、统计、图表和 LaTeX 输出能力。

## 平台定位

HPM-DT 不是一个算法仓库，不是几个 Python 脚本，也不是单个论文项目。它的定位是：

> 高功率微波任务级数字孪生 CAE 平台。

当前项目仍坚持安全研究边界：核心求解模型均为归一化、公开、可复现的数值模型；平台可以把用户输入的阵元功率作为实测标定元数据并输出残差/不确定度，但不输出真实毁伤概率、真实作用距离或真实器件阈值，不替代 CST/HFSS/COMSOL 全波仿真。

## Scene First 原则

V2.0 的产品路线必须从 Verification First 调整为 Scene First。打开软件时，用户应直接进入三维场景工作台，看到阵列、目标、观察面、反射面、保护区、求解入口和结果图层；验证中心只能作为 `工具 -> 可信度验证` 的高级功能，用于发文、报告、验收和结果复核。

后续新增功能必须优先服务场景和任务：它让用户能建立什么新场景、运行什么新任务、查看什么新结果。只有在这个主链路接通后，才继续扩展验证指标、成熟度报表或后台审计。

## 长期八层架构

### 第一层：物理建模层 Physics Layer

负责阵列、传播、多径、孔缝、腔体、近场、远场、材料等基础物理与降阶模型。

### 第二层：感知层 Perception Layer

负责 DOA、MUSIC、ESPRIT、CNN、Transformer、目标识别和感知不确定度。

### 第三层：防护层 Protection Layer

负责 MVDR、LCMV、宽零陷、鲁棒波束形成、动态防护和防护策略审计。

### 第四层：控场层 Field Control Layer

负责区域赋形、多目标控场、RIS、协同阵列和鲁棒优化。

### 第五层：效应层 Effect Layer

负责剂量、风险、概率、敏感度和效应数字孪生代理模型。

### 第六层：可信度层 Verification & Validation Layer

负责解析验证、文献复现、不确定度、适用性、后端退化审计和可信度评分。

### 第七层：CAE 平台层 Workbench Layer

负责项目、建模、求解、结果、报告、数据库、版本管理和工程资产管理。

### 第八层：论文生产层 Publication Layer

负责实验、统计、图表、LaTeX、IEEE 模板和论文自动生成。

## 最终用户流程

目标形态是用户打开平台后完成如下闭环：

```text
建立场景
↓
配置阵列与系统
↓
运行任务
↓
查看场分布与风险
↓
导出论文
```

## 当前版本在 North Star 中的位置

V2.0A 是可信度层的阶段里程碑。它证明平台可以把既有算法结果放入可验证、可复现、可审计的 V&V 框架，但它不是项目的最高层目标，也不应作为默认首页。

最新 V2.0B 增量把 Workbench Layer 继续向科研级 CAE 平台推进：前端默认入口已经切换为“场景编辑”，首屏固定呈现 `场景 / 对象 / 任务 / 结果`，并把可信度验证移动到工具入口；三维工作台已经具备本地 Three.js 视口、受控对象/材料编辑、材料代理审计、按钮式受控几何变换控件、求解任务队列、本地后台 worker 预览、暂停/恢复检查点、任务取消/重试/审计协议、结果图层与 x/y 剖面交互曲线、V&V 适用性诊断区、求解结果档案、对象级指标列表/三维徽标、工程快照落盘/恢复、快照差异对比和统一工程资产台账。统一台账把任务、结果、快照、导入标定桥接和 `MAT-AUDIT-001` 材料代理审计汇总为 JSON/CSV/SQLite 可审计资产索引，并把 JOB 操作日志、任务、结果、快照和数据库 manifest 展开为 SQLite 表，提供类型/关键字筛选、资产审计、数据库审计、库表浏览、资产谱系追溯和命名规范审计报告；谱系追溯把 `SNP/JOB/SOL` 节点、同 `scene_hash` 基线、任务生成结果、重试派生和事件派生关系输出为 `lineage.json`/`lineage.csv`，用于支撑“项目-建模-求解-结果-验证-版本管理”的平台闭环，而不是孤立的 UI 展示。

后续所有新增功能都应回答三个问题：

1. 它是否让用户能建立、运行或分析一个新的场景/任务？
2. 它是否增强八层架构中的某一层，并保持层间接口清晰？
3. 它是否能被验证、复现、审计，并最终进入论文或项目申报材料？
